lynx과 w3m

: 텍스트 기반의 웹 브라우저로서 리눅스와 유닉스 기반 시스템에서 사용되는 도구이다. 터미널 환경에서 웹 페이지를 탐색하고 볼 수 있는 기능을 제공한다. 서버나 네트워크 상태를 확인하거나 웹 페이지의 구조를 파악할 때 유용하다. 또, 리눅스 서버에서 웹 페이지를 렌더링하지 않는 경우에도 웹 사이트에 접근하여 정보를 확인할 수 있는 간단하고 가벼운 대안으로 사용될 수 있다.

lynx

: 가장 오래된 텍스트 기반 웹 브라우저 중 하나로, 텍스트 환경에서 동작하며 사용자가 키보드를 통해 링크를 따라 이동하고 웹 페이지를 읽을 수 있도록 해준다. 기본적인 HTML을 표시할 수 있으며, 이미지나 CSS같은 고급 기능은 지원하지 않는다. lynx는 명령어 기반 인터페이스를 사용하여 웹 페이지를 탐색하고 링크를 따라 이동할 수 있다.


w3m

: lynx와 유사한 목적을 가진 텍스트 기반 웹 브라우저로, lynx보다 좀 더 향상된 기능을 제공하며 이미지 표시, 테이블 지원, 쿠키 관리, 탭 지원 등과 같은 고급 기능을 제공한다. 이전 페이지와 다음 페이지로 이동하고, 북마크를 추가하고, 검색 기능을 사용할 수 있으며, 커스터마이즈할 수 있는 키 바인딩 등의 기능을 제공한다. w3m은 명령어 기반 브라우저인 동시에 이미지 뷰어로서도 사용할 수 있다.

service

: 리눅스와 유닉스 기반 운영체제에서 시스템 서비스를 관리하는 명령어이다. service 명령어를 사용하여 시스템의 서비스를 시작, 중지, 다시 시작, 확인 및 구성할 수 있다.

service 주요 기능
  • 서비스 관리 : 시스템에서 실행 중인 서비스룰 시작, 중지, 다시 시작할 수 있다. (예 -> service apache2 start : Apache 웹 서버 상태 확인 / service sshd stop : SSH 서비스 중지)
  • 서비스 상태 확인 : 시스템에서 실행 중인 서비스의 상태를 확인할 수 있다. (예 -> service apache2 status : Apache 웹 서버 상태 확인 / service sshd status : SSH 서비스의 상태 확인)
  • 부팅 시 자동 실행 설정 : 특정 서비스가 시스템 부팅 시 자동으로 시작되도록 설정할 수 있다. (예 -> service apache2 enable : Apache 웹 서버가 부팅 시 자동으로 시작되도록 설정 / service sshd disable : SSH 서비스 부팅 시 자동으로 시작되지 않도록 설정)
  • 서비스 구성 파일 관리 : 서비스 구성 파일을 관리하는 데 사용될 수 있다. (예 -> service apache2 configtest : Apache 웹 서버 구성 파일의 유효성 테스트 / service sshd reload : SSH 서비스 구성 다시 로드)

git과 SVN

: git과 SVN은 형상관리 툴 종류 중 하나로 형상관리 툴(소프트웨어 버전 관리 툴)이란소스를 버전 별로 관리할 수 있어 개발할 때 실수로 소스를 삭제하거나, 수정하기 이전으로 돌아가야되는 경우 유용하게 사용되는 툴이다. 코드를 병합하거나 수정된 소스를 추적하는 데에도 쓰이는 형상관리 툴의 종류에는 Client/Server타입, 분산저장소 타입, Folder 공유 타입이 있는데, git은 분산저장소 타입에 해당하며, SVN은 Client/Server 타입에 해당한다.

git

: 매우 빠른 속도의 분산형 저장소로 SVN보다 많은 기능을 지원하는 대신 익숙해지기에 더 많은 시간이 필요하다. 개발자가 자신만의 commit history를 가질 수 있고, 개발자와 서버의 저장소는 독립적으로 관리가 가능하다. commit한 내용에 실수가 있더라도 바로 서버에 영향을 미치지 않고, 개발자는 마음대로 commit하다가 자신이 원하는 순간에 서버에 변경 내역을 보낼 수 있으며, 서버의 통합 관리자는 관리자가 원하는 순간에 각 개발자의 변경 내역을 가져올 수 있다.


SVN(Subversion)

: CVS를 대체하기 위해 만들어져 현재까지 두루 사용되는 형상관리 툴로, 보통 대부분의 기능을 완성해놓고 소스를 중앙 저장소에 commit(중앙 저장소에 해당 기능을 공개)한다. 개발자가 자신만의 version history를 가질 수 없고, commit한 내용에 실수가 있을 시 다른 개발자에게 바로 영향을 미치게 된다는 단점이 있다.

branch

: branch란 독립적으로 어떤 작업을 진행하기 위한 개념이다. 필요에 의해 만들어지는 각각의 브랜치는 다른 브랜치의 영향을 받지 않기 때문에 여러 작업을 동시에 진행할 수 있다. 만들어진 브랜치는 다른 브랜치와 병합(Merge)함으로써 작업한 내용을 다시 새로운 하나의 브랜치로 모을 수 있다.
기본 브랜치(main 또는 master branch)는 프로젝트의 안정적인 버전을 관리하는 브랜치로, 일반적으로 제품 릴리스나 안정 버전의 배포에 사용된다.
새로운 브랜치는 기본 브랜치에서 분기하여 생성할 수 있다. 이를 통해 별도의 작업 영역을 만들고, 새로운 기능 개발, 버그 수정, 실험 등을 진행할 수 있다. 그리고 새로운 브랜치는 기본 브랜치의 상태를기반으로 생성되며, 이후에는 독립적으로 변경 사항을 관리한다.
branch 생성 명령어 : git branch [브랜치 명] / branch 이동 명령어 : git checkout [브랜치 명]

stash

: stash는 변경 사항을 일시적으로 저장하고 브랜치를 전환하거나 다른 작업을 수행할 때 유용한 도구로, 변경 사항을 커밋하지 않고 작업 영역을 정리하며, 필요한 경우에 변경 사항을 다시 적용하여 작업을 이어갈 수 있다. 버전 관리 시스템에서 제공하는 유용한 기능 중 하나이며, 개발 작업의 효율성을 높이는 데 도움이 된다.

stash 기능 및 명령어
  • 변경 사항 보관 : 현재 작업 중인 브랜치에서 변경한 내용을 일시적으로 저장한다. 변경 사항을 커밋하지 않고 임시로 보관하여 나중에 필요할 때 다시 적용할 수 있다.
  • 작업 영역 정리 : stash를 사용하면 작업 중인 브랜치에서 변경 사항을 일시적으로 보관하여 작업 영역을 깨끗하게 유지할 수 있다.
  • stash 명령어 : git stash 명령어를 통해 변경 사항을 stash할 수 있다.
  • stash 적용 명령어 : git stash apply 명령어를 통해 저장된 변경 사항을 필요할 때 다시 적용할 수 있다.

merge

: 병합(merge)은 버전 관리 시스템에서 브랜치 간의 변경 사항을 통합하는 과정이다. 주로 개발자들이 동시에 별도의 브랜치에서 작업을 진행한 후, 변경 사항을 하나의 브랜치로 통합하고자 할 때 사용된다. merge를 통해 다수의 작업 내용을 하나로 합치거나 메인 브랜치(기본 브랜치)로 변경 사항을 병합할 수 있다.

merge 종류
  • 3-way merge : 가장 일반적으로 사용되는 방법으로, 3가지 버전(공통 기준, 현재 브랜치의 변경, 병합 대상 브랜치의 변경)을 비교하여 변경 사항을 통합한다. 충돌이 발생하는 경우에는 수동으로 해결해야 한다.
  • Octopus merge : git과 같은 분산 버전 관리 시스템에서 지원되는 특별한 병합 방법으로, 3개 이상의 브랜치를 병합하는 경우에 사용된다.
  • Fast-forward merge : 기준 브랜치와 병합할 브랜치가 존재하며, 병합할 브랜치가 기준 브랜치에 비해 앞선 커밋을 가지고 있는 경우에 사용된다. 변경 사항이 별도로 생성되지 않고, 기준 브랜치를 단순히 앞선 커밋으로 이동시킨다.

conflict

: 충돌(conflict)은 병합 작업 중에 발생하는 상황으로 2개 이상의 변경 사항이 동일한 부분을 수정한 경우에 발생한다. 주로 버전 관리 시스템에서 브랜치 병합 작업을 수행할 때 발생하며, 충돌을 해결해야 정확한 통합이 가능해진다.
충돌 해결은 수동으로 이루어지며 주로 소스 코드 파일에서 이루어지지만, 기타 텍스트 파일에서도 발생할 수 있다. 버전 관리 시스템은 충돌 해결을 위해 도구를 제공하는데 이는 충돌 부분을 시각적으로 표시하고, 변경 사항을 비교하여 충돌을 해결하는 데 도움을 준다.

Jenkins

: CI/CD를 위한 대표적인 도구로, 빌드, 테스트, 배포 프로세스를 자동화하여 소프트웨어 품질 향상과 개발 생산성 향상에 도움을 주는 도구이다. 소스 변경 사항을 프로젝트에 지속적으로 통합하여 코드의 오류를 조기에 발견하고 빠른 피드백을 통해 안정적인 릴리즈를 지속적으로 제공할 수 있다.

Jenkins 이점
  • 지속적 통합 및 지속적 제공 : 확장 가능한 자동화 서버인 Jenkins는 간단한 CI 서버로 사용하거나 모든 프로젝트의 지속적인 전달 허브로 전환할 수 있다.
  • 쉬운 설치 : 독립형 Java 기반 프로그램으로 Windows, Linux, macOS 및 기타 Unix 계열 운영체제용 패키지와 함께 즉시 실행할 수 있다.
  • 쉬운 구성 : 즉석 오류 검사 및 기본 제공 도움말이 포함된 웹 인터페이스를 통해 쉽게 설정하고 구성할 수 있다.
  • 플러그인 : 업데이트 센터에 있는 수백 개의 플러그인을 통해 지속적 통합 및 지속적 제공 도구 체인의 거의 모든 도구와 통합된다.
  • 확장 가능 : 플러그인 아키텍처를 통해 확장할 수 있으므로 할 수 있는 작업에 거의 무한한 가능성을 제공한다.
  • 분산 : 여러 시스템에 작업을 쉽게 배포할 수 있으므로 여러 플랫폼에서 빌드, 테스트 및 배포를 더 빠르게 진행할 수 있다.

ELK 스택(Elasticsearch, Logstash, Kibana)

: ELK 스택은 Elasticsearch, Logstash, Kibana의 세 가지 인기 있는 프로젝트로 구성된 스택을 의미하는 약어이다. 사용자에게 모든 시스템과 애플리케이션에서 로그를 집계하고 이를 분석하며 애플리케이션과 인프라 모니터링 시각화를 생성하고, 빠르게 문제를 해결하며 보안 분석할 수 있는 능력을 제공한다.

Elasticsearch

: Apache Lucene에 구축되어 배포된 검색 및 분석 엔진이다. Elasticsearch는 자동으로 원래 문서를 저장하고 클러스터의 인덱스에 문서에 대한 검색 가능한 참조를 추가하며, Elasticsearch API를 사용해 문서를 검색하고 조회할 수 있다.

Elasticsearch 이점
  • 신속한 가치 실현 : 간단한 REST 기반 API, 간단한 HTTP 인터페이스를 제공하고 스키마가 없는 JSON 문서를 사용해 다양한 사용 사례에서 쉽게 시작하고 빠르게 애플리케이션을 구축할 수 있다.
  • 고성능 : 분산 성질로 인해 대량 볼륨의 데이터를 병렬로 처리할 수 있어 쿼리에 최고의 일치 항목을 빠르게 찾을 수 있다.
  • 무료 도구 및 플러그인 : Elasticsearch는 Kibana가 통합되어 제공되고, Beats 및 Logstash와의 통합도 제공하여 소스 데이터를 쉽게 전환하고 Elasticsearch 클러스터에 로드할 수 있다. 다수의 오픈 소스 Elasticsearch 플러그인을 사용하여 애플리케이션에 풍부한 기능을 추가할 수도 있다.
  • 실시간에 가까운 운영 : 데이터 읽기 및 쓰기와 같은 Elasticsearch 운영은 완료까지 보통 1초도 걸리지 않아 실시간에 가깝게 사용할 수 있다.
  • 쉬운 애플리케이션 개발 : Java, Python, PHP, JavaScript, Node.js, Ruby 및 기타 여러 다양한 언어에 대한 지원을 제공한다.

Logstash

: 다양한 소스로부터 데이터를 수집하고 전환하여 원하는 대상에 전송할 수 있도록 하는 오픈 소스 데이터 수집 도구이다. 사전 구축된 필터와 200개가 넘은 플러그인에 대한 지원으로 Logstash는 사용자가 데이터 원본이나 유형에 관계없이 데이터를 쉽게 수집할 수 있도록 도와준다.

Kibana

: 로그 및 이벤트 검토에 사용하는 데이터 시각화 및 탐색 도구이다. 사용하기 쉽고 대화형 차트와 사전 구축된 집계 및 필터, 지리 공간적 지원을 제고아고 이를 사용해 Elasticsearch에 저장된 데이터를 시각화할 때 원하는 선택을 할 수 있다.


ELK 스택 작동 방식 :
Logstash는 데이터를 수집 및 변환하고 올바른 대상으로 전송한다.
Elasticsearch는 수집된 데이터를 인덱싱하고, 분석하고, 검색한다.
Kibana는 분석 결과를 시각화한다.
